사용자의 요구사항

시스템명 : 통합보안시스템
요구사항 ID : SECU_분야1, 분야2 ...
목적 : AI 동화 창작 시스템을 운영을 보조/보호하는 보안시스템

외부 위협 방어 / 침입탐지 시스템 관련

사용 예정 플랫폼
 - IP 상태 정보, 경고 횟수, 차단 TTL을 관리하기 위한 상태 저장소(Redis)
 - 빠른 처리를 위한 kafka
 - 내부 탐지용 wazuh
 - 외부 탐지용 spiderfoot
 - 실제 컨트롤 방화벽 nftables
 - 모니터링용 wazuh, opensearch
 - 정책 결정 및 명령 실행용 Open policy agent



(판단 규칙) 판단에는 아래 빈도+행위 기반 규칙이 적용된다. 기본적으로 자동차단 관리 기능에 탑재되며, 관리자도 가급적이면 아래 규칙을 참고하여 차단을 하여야 한다. 다만 1일 이상 차단시 관리자의 재량에 따라 처벌이 가능하다.
빈도 : 빈도 기준은 시스템 부하 및 오탐률을 고려하여 운영 환경에 따라 조정 가능하도록 설계한다. 
행위 : 경고 횟수는 빈도 기반 규칙을 위반할 때마다 쌓이며, 아래 항목을 기준으로 차단을 한다.
 - 1회 ~ 5회 : 10분 간 접속 차단
 - 6회 ~ 9회 : 20분 간 접속 차단
 - 10회~19회 : 30분간 접속 차단
 - 20회~29회 : 1시간 접속 차단
 - 30회~49회 : 3시간 접속 차단
 - 50회 이상 : 1일 접속 차단
 - 평판이 warning 일때 : 1시간 접속 차단
 - 평판이 malicious 일때 : 3시간 접속 차단


정책 판단 시 우선순위는 다음과 같다.
1. 관리자 수동 판단
2. 내부 행위 기반 경고 횟수
3. 외부 평판 점수


판단 예외 규칙
 - 평판 기반 차단은 기본적으로 1회성 차단을 원칙으로 하며, 지속 차단은 내부 행위 기반 경고 횟수와 결합될 경우에만 적용한다.
 - 1회성 차단이란, 해당 요청에 대해서만 즉시 차단 응답을 반환하며, 방화벽(nftables)에는 규칙을 등록하지 않는다.
 - 위 규칙의 예외대상은 관리자이다.


SEC-ALT-01. 자동 차단 관리 기능
개요 : 이 기능은 침입탐지 및 차단관리 기능을 기술한것이며, 이 기능은 시스템상 아래 규칙에 의해 작동되는 자동 처리와 관리자가 직접 관리하는 수동 처리으로 나뉜다. 자동처리는 시스템이 평판 API의 위험도를 분석해 몇몇 악성행위를 자동으로 탐지하고 아래 기준을 통해 자동으로 처리하는 방법이다. 수동처리는 시스템이 자동으로 탐지되지 아니하며, 악성행위를 보안관리자가 직접 IP를 경고하거나 차단하는 처리 방법이다. 

로직 : 클라이언트 접속 -> Wazuh / IDS 이벤트 수집 및 kafka를 이용하여 전달 -> Redis와 spiderfoot를 통한 평판 조회 -> OPA 정보 판단 및 구분 -> nftables를 이용한 차단 실행

우회로직 : 
만약 규칙에 부합하지 않는다면
 - 판단 결과를 이벤트로 기록한 후 추가 차단 조치는 수행하지 않는다.



SEC-ALT-02. 수동 차단 관리 기능
개요 : 시스템이 자동으로 처리가 불가능하거나 제대로 작동하지 않는 경우, 이때 발생하는 악성행위를 보안관리자가 직접 IP를 경고하거나 차단하는 처리 방법이다. 보안 대시보드를 기반으로 관리자가 직접 차단을 하거나 해제할 수 있다. 위 자동 처리 과정에서 오차단을 통해 접속 민원이 있을 경우나 기타 상황을 대비하여 차단 해제도 되어야한다.


로직 : 클라이언트의 접속기록 체크 -> 대시보드를 이용해 악성 유저로 의심되는 사용자 IP를 추적 -> 관리자가 판단후 위 규칙대로 차단 요청 및 이벤트 발생 -> IP 블랙리스트 관리 기능으로 연계한다


우회로직 : 
관리자가 판단후 무해하다 판단할 경우
 - 추가조치를 하지 않는다.

차단으로 인해 클라이언트가 명백한 사유로 IP 차단 해제를 요청한다면
 - 관리자가 IP 차단을 해제한다.

IP가 이미 블랙리스트에 있다면
 - 자동 차단은 갱신차단되며, 수동 차단은 관리자의 판단에 따른 조치이므로, 중복 패널티는 적용하지 않는다.



SEC-ALT-03. IP 블랙리스트 관리 기능
개요 : 위 자동 차단 관리 기능과 수동 차단 관리 기능에서 나온 IP를 저장하고 이를 리스트에 올려서 각 IP에 대한 차단조치를 실질적으로 하게 된다. 이를 위해 Redis으로 형식을 정해서 저장하여야 하며, 경고횟수에 따른 조치도 적절하게 취해야한다. 테이블에 아래 양식대로 저장해야한다.

로직1 : 탐지기능이나 차단관리 기능에서 나온 정보를 이벤트 형식으로 kafka를 통해 전달 한다 -> 상태정보 DB(Redis)에 각 정보를 등록해둔다 -> OPA를 통해 정보에 따른 조치를 정한다. ->  nftables 적용을 한다. -> 로직이 종료된다.

로직2 : 차단 해지 요청을 받는다 -> 상태정보 DB(Redis)에 목록에 삭제한다 -> OPA를 통해 지시한다. ->  nftables를 이용해 시스템이 방화벽에 등록된 ip 대역을 해제한다. -> 로직이 종료된다.

양식
{
  "source": "spiderfoot",
  "ip": "185.XX.XX.12",
  "threat": "malicious",
  "confidence": 0.82
}


SEC-ALT-04. IP 평판 조회 및 판별 기능
개요 : 이건 외부 API를 통해 IP의 평판을 조회하고 이를 대시보드나 SIEM에 알리도록 하는 기능이다. 주로 IP주소를 대입하면 외부 API의 데이터베이스에서 조회하여 IP의 위험도와 confidence를 SEC-ALT-01, SEC-ALT-02에 알리는 보조기능이다.

로직 : SEC-ALT-01, SEC-ALT-02에서 가져온 IP를 spiderfoot에 대입한다 -> IP의 위험도를 수집한다. -> 수집한 데이터를 SEC-ALT-01, SEC-ALT-02에 아래 양식대로 전달한다.

양식 예시
{
  "source": "spiderfoot",
  "ip": "185.XX.XX.12",
  "threat": "malicious",
  "confidence": 0.82
}

조회 IP 별 구분 기준
1. 악성 : malicious
2. 주의 : warning
3. 보통 : normal
4. 안전 : safe
5. 모름 : unknown


우회로직 :
 - 조회할 IP가 없는 경우
IP 재요청을 한 뒤에 상태 DB에 이벤트로 남긴다.




비기능적 요구사항
- 이벤트 탐지부터 정책 판단까지는 10초 이내 완료
- 차단 적용은 최초 탐지 후 최대 15초 이내 완료
